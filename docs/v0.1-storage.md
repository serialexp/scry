# v0.1 — Storage layer

> **Status: ✓ complete.** Sealed by `scripts/smoke.sh` (the scripted
> exit criterion) and pinned in [`decisions.md § D-030`](./decisions.md).
>
> Minimum viable storage layer: a dummy record goes in via the wire
> protocol, lands in the WAL, becomes a parquet block in Garage, and
> shows up in the catalog. No signal-specific code, no query, no
> distributed concerns. The smallest thing that proves the architecture
> in [`ARCHITECTURE.md`](./ARCHITECTURE.md) actually works end-to-end.

This document is **scoped to v0.1.** It deliberately defers everything
non-essential to validating the storage pipeline. Cross-references to
`ARCHITECTURE.md` indicate "this is settled, don't re-decide here."

## What we're building

```
                                      ┌── scry-objstore ──── Garage (docker)
                                      │
noise-spewer ──[wire]──→ noise-sink ──┼── scry-wal       ──── ./wal/dummy/
                                      │
                                      ├── scry-block     ──── (in-RAM builder)
                                      │
                                      └── scry-catalog   ──── ./catalog.sqlite
```

The noise-sink stops being a counter and becomes a writer. A new
`scry-list` bin (or a `--mode reader` flag) bootstraps an empty catalog
from the Garage bucket and prints what it finds.

## Non-goals (v0.1 is small on purpose)

Anything not in the diagram above is **out of scope**. Specifically:

- **No signal split.** Single `dummy/` prefix. No `metrics/`, `logs/`,
  `traces/`, `profiles/`. The signal-aware paths come back when their
  signals come back, in later milestones.
- **No multi-bucket / sealing / auto-provisioning.** One bucket
  (`scry-dev` in Garage). The whole bucket-pool design from
  `ARCHITECTURE.md` is dormant.
- **No Valkey.** One writer, one catalog. Multi-instance convergence
  doesn't apply yet.
- **No compaction, no retention.** Blocks accumulate. We'll address
  cleanup when there's something to clean up.
- **No postings index.** Metrics-only sidecar; doesn't exist yet.
- **No catalog snapshot.** Bootstrap is "walk the bucket" only.
- **No query engine.** The "reader" reads the catalog and prints
  block metadata; it does not load parquet content. DataFusion comes
  in v0.2+ alongside the first real signal.
- **No resource-isolation runtime split.** One Tokio runtime. The
  named-runtime / token-bucket discipline from D-027 lands when the
  signals do.
- **No agent-side spool semantics change.** noise-spewer keeps doing
  what it does today; we still accept its at-least-once delivery and
  rely on its `(agent_id, batch_id)` to dedup later.

## The dummy record

A single binschema type, defined in `proto/ingest.schema.json` next to
the wire types:

```json
"DummyRecord": {
  "sequence": [
    { "name": "ts_unix_nano", "type": "uint64" },
    { "name": "key",          "type": "string", "kind": "length_prefixed", "length_type": "uint16", "encoding": "utf8" },
    { "name": "value",        "type": "bytes",  "kind": "length_prefixed", "length_type": "uint32" }
  ]
}
```

A `DummyBatch { records: Vec<DummyRecord> }` envelope wraps a batch.
Generated by `scripts/gen-proto.sh` into `crates/proto/src/generated.rs`,
re-exported as `scry_proto::{DummyBatch, DummyRecord}`.

The schemas live in one file because `binschema generate` takes a single
`--schema` argument; splitting them at the file level would require
extending the codegen for no real benefit. The "wire vs durable"
distinction is documentary, not structural.

The WAL payload encoding is the binschema-encoded `DummyRecord`. The
parquet schema mirrors it:

| column         | type     | parquet encoding              |
|----------------|----------|-------------------------------|
| `ts_unix_nano` | `uint64` | `PLAIN`, sorted ascending     |
| `key`          | `utf8`   | `DELTA_BYTE_ARRAY` (or dict)  |
| `value`        | `binary` | `PLAIN` + zstd column compr.  |

Block is sorted by `ts_unix_nano` on close. Row groups sized to ~1 MiB
compressed (per `ARCHITECTURE.md § The block builder`).

## What gets transmitted on the wire

The existing `Batch` frame is reused as-is. For v0.1 the spewer is
modified (small patch, post-doc) to:

- emit `signal = 0xFE` ("dummy" — a value outside the
  `Signal::{Metrics,Logs,Traces,Profiles}` enum, added as
  `Signal::Dummy = 0xFE` in `constants.rs` for v0.1 only;
  `0xFF` is taken by `SIGNAL_ALL` in `FlowControl`),
- pack the payload as a length-prefixed list of `DummyRecord`s, zstd
  level 3 — same envelope as today.

When real signals come online, `Dummy` goes away. It's not a stable
part of the protocol; the wire schema doesn't need a version bump
because the discriminator value is just "an unrecognised signal" until
v0.1.

## Crate skeleton

Four new crates, each owning one concern. Composition lives in the
existing `noise-sink` for v0.1 (the eventual `scry-server` crate is a
v0.2 concern).

### `scry-objstore`

**Owns:** object-store configuration, a `Box<dyn object_store::ObjectStore>`
factory, and the small set of operations the rest of the storage layer
needs.

**Depends on:** `object_store` (apache/arrow-rs), `bytes`, `anyhow`.

```rust
pub struct ObjStoreConfig {
    pub endpoint: String,         // http://127.0.0.1:3900
    pub region:   String,         // garage
    pub bucket:   String,         // scry-dev
    pub access_key_id:     String,
    pub secret_access_key: String,
    pub path_style: bool,         // true for Garage; false for AWS-style
}

pub fn open(cfg: &ObjStoreConfig) -> anyhow::Result<Arc<dyn ObjectStore>>;
```

Rationale: we don't write S3 retry / multipart / signing code, and
`object_store` already abstracts local filesystem too — useful for
unit tests that don't want to spin up Garage.

**Conditional PUT note.** Garage 1.0.x silently ignores
`If-None-Match: *`, so we don't expose a `put_if_absent` helper in
v0.1. Block paths are addressed by UUID v7 and a single writer never
issues two PUTs to the same path, so we don't need it for correctness.
When we move to real S3 / R2 (or Garage gains support), the helper
goes back in as a thin wrap around `PutMode::Create`.

### `scry-wal`

**Owns:** the on-disk WAL described in `ARCHITECTURE.md § The WAL`.
Segment writer, segment reader, replay iterator, upload-marker bookkeeping.

**Depends on:** `tokio` (fs + io-util), `crc32fast`, `anyhow`.

```rust
pub struct Wal { /* per-signal dir, current segment, seq counter */ }

impl Wal {
    pub fn open(dir: &Path, signal: &str) -> anyhow::Result<Self>;
    pub async fn append(&mut self, payload: &[u8]) -> anyhow::Result<()>;
    pub async fn rotate(&mut self) -> anyhow::Result<SegmentId>;
    pub fn replay(&self) -> impl Iterator<Item = anyhow::Result<Vec<u8>>>;
    pub fn mark_uploaded(&mut self, up_to: SegmentId) -> anyhow::Result<()>;
}
```

Generic over payload bytes — `scry-wal` does not know about
`DummyRecord` or any other type. The framing (`[len:u32][crc32:u32]
[payload]`) is its only schema commitment.

### `scry-block`

**Owns:** the in-memory block builder (arrow `RecordBatch` builders,
zstd-3 parquet serialisation, sidecar serde) and a block reader that
opens a parquet object via `object_store` and yields `RecordBatch`es.

**Depends on:** `arrow`, `parquet`, `serde`, `serde_json`, `scry-objstore`,
`anyhow`.

```rust
pub struct BlockBuilder { /* arrow builders, row count, byte est */ }

impl BlockBuilder {
    pub fn new(signal: &'static str, writer_id: Uuid) -> Self;
    pub fn append(&mut self, rec: &DummyRecord) -> anyhow::Result<()>;
    pub fn should_close(&self, now: Instant) -> bool;
    pub async fn finish_and_upload(
        self,
        store: &dyn ObjectStore,
    ) -> anyhow::Result<BlockMeta>;
}

pub struct BlockMeta {
    pub uuid:        Uuid,
    pub signal:      String,
    pub writer_id:   Uuid,
    pub ts_min:      u64,
    pub ts_max:      u64,
    pub row_count:   u64,
    pub byte_size:   u64,
    pub schema_version: u32,
}
```

Sidecar is `<path>.meta.json`; `BlockMeta` serialises to it. The
fingerprint bloom + per-column min/max bits from `ARCHITECTURE.md` are
deferred — there's no label space for the dummy record, and predicate
pushdown isn't exercised in v0.1. The fields stay in the catalog schema
as nullables so we don't migrate later; the writer just leaves them
null.

For v0.1 the builder owns one block at a time (no `(signal, day)` map),
because there's one signal. The day-rollover trigger is implemented
but in v0.1 it never fires (all data is "now").

### `scry-catalog`

**Owns:** the SQLite-backed catalog. Embedded schema (verbatim from
`ARCHITECTURE.md § The catalog § Schema`), prepared statements, the
operations writer and reader need.

**Depends on:** `rusqlite` (bundled feature, no system sqlite), `anyhow`.

```rust
pub struct Catalog { /* rusqlite Connection */ }

impl Catalog {
    pub fn open(path: &Path) -> anyhow::Result<Self>;       // creates schema if missing
    pub fn insert_block(&self, m: &BlockMeta) -> anyhow::Result<()>;
    pub fn list_blocks(&self) -> anyhow::Result<Vec<BlockMeta>>;
    pub fn reconcile_from_bucket(
        &self,
        store: &dyn ObjectStore,
    ) -> anyhow::Result<usize>;  // walk + upsert sidecars; returns count
}
```

Rationale for `rusqlite` over `sqlx`: simpler, synchronous, no
compile-time SQL checks needed for a four-table catalog, no `.sqlx/`
cache to commit. We can swap later if we want; the public API doesn't
expose the choice.

### Composition (in `noise-sink`)

For v0.1 the sink grows three fields: `Wal`, `BlockBuilder`,
`Arc<dyn ObjectStore>`, plus a `Catalog`. Per-batch:

1. Decompress payload (already done today).
2. Decode as `Vec<DummyRecord>` (new).
3. For each record: `wal.append(binschema_bytes)` then
   `builder.append(rec)`.
4. After the batch: if `builder.should_close(now)`, call
   `finish_and_upload`, then `catalog.insert_block`, then
   `wal.mark_uploaded(...)`.
5. Send `BatchAck`.

Crash recovery on sink startup: replay the WAL into a fresh builder,
then resume.

## Garage (docker)

Single-node, all on the dev machine. Lives under `docker/garage/`:

```
docker/garage/
  docker-compose.yml
  garage.toml         # data dir, RPC bind, S3 endpoint
  .env.example        # GARAGE_ADMIN_TOKEN, ACCESS_KEY, SECRET_KEY
  init.sh             # one-shot: layout, create bucket, create key
```

Defaults:

- S3 endpoint: `http://127.0.0.1:3900`
- Region: `garage`
- Bucket: `scry-dev`
- Access key / secret key written into `.env` (gitignored), example in
  `.env.example`.

Driven by:

```
scripts/dev-garage-up.sh    # docker compose up -d + init.sh if needed
scripts/dev-garage-down.sh  # docker compose down -v
```

The integration test reads the same `.env` so dev and CI can share the
container.

## v0.1 exit criteria — met by `scripts/smoke.sh`

The scripted criterion is `scripts/smoke.sh`. It is what now decides
whether v0.1 is done on any given checkout. The sequence:

1. `scripts/dev-garage-up.sh` (prerequisite — the smoke script
   refuses to run without `docker/garage/.env`).
2. Empty the dev Garage bucket so the post-condition is unambiguous.
3. Start the modified `noise-sink` with `--storage --wal-dir
   --catalog`.
4. Run `noise-spewer --signals dummy --rate 400 --max-batches 2000
   --duration 30s`. `--max-batches` makes the record count exact;
   `--rate × --duration` is off-by-one in practice (the spewer trims
   one interval tick on graceful exit), and the smoke script's
   assertion needs an integer match.
5. SIGINT noise-sink (graceful → final flush).
6. Run `scry-list` against an *empty* catalog file. It calls
   `catalog.reconcile_from_bucket(...)` and prints one line per
   block plus a `# total rows=N bytes=N` trailer.
7. Assert: `total_rows == max_batches × 256` (every DummyBatch
   carries exactly 256 records — `crates/noise-spewer/src/gen.rs ::
   render_dummy`) **and** `block_count >= 1`.

Default parameters (`BATCHES=2000`, `RATE=400`, `DURATION_SECS=30`)
give 512,000 records → one ~24 MB parquet block. Override via env vars
for stress runs (`BATCHES=20000 RATE=4000 scripts/smoke.sh`).

## Open questions parked for v0.2

These are real, just not v0.1:

- The fingerprint bloom in the catalog `fingerprint BLOB` column — what
  hash, what false-positive rate. (Decided when the first signal with
  labels lands.)
- WAL replay across the `(signal, day)` boundary — v0.1 has no day
  boundary because there's only "now."
- How signal-specific block builders share `scry-block` — likely a
  trait per signal that produces a `RecordBatch`; details when
  signal #2 arrives.
- Whether `scry-server` is its own crate or a binary inside
  `scry-storage` — defer until we see how the wiring shapes up.
